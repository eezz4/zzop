Using zzop

zzop ships as zzop-mcp, a single Node-free native binary: no Node.js, no npm, nothing to compile. Run it as a CLI over subcommands (zzop-mcp analyze . — no code, ESLint-style) or point an MCP client at zzop-mcp mcp. This page is the CLI — if you're building a tool on top of zzop instead, see the SDK reference.

Command line

Install & run the binary

Three ways to get it — no source checkout or toolchain required for any of them:

  • Claude Code — /plugin marketplace add eezz4/zzop, then /plugin install zzop@zzop. The plugin's .mcp.json runs zzop-mcp mcp for you.
  • Claude Desktop — the one-click .mcpb bundle (drag-and-drop install); it carries the platform binary. See packaging/README.md.
  • CLI / CI — download the zzop-mcp-<platform>[.exe] binary from GitHub Releases (5 platforms) and put it on PATH as zzop-mcp.
Analyze a tree
zzop-mcp analyze .        # analyze this tree, print JSON findings
zzop-mcp cross ../api ../web   # cross-layer join across 2+ trees
zzop-mcp mcp               # serve MCP over stdio for Claude Code/Desktop

See packages/mcp/README.md for the full CLI reference. (The npm distribution — the JS CLI @zzop/cli and the napi addon @zzop/native, installed via npm i -D @zzop/cli / npx zzop — was removed 2026-07-20; zzop-mcp is its full Node-free replacement.)

CLI config

zzop.config.jsonc

zzop.config.jsonc is JSON with comments and trailing commas allowed. zzop-mcp analyze <path> auto-discovers one at the tree root; cross/endpoint load one explicitly via --config <path>. There is no scaffolding command to generate a starter file (the JS CLI's zzop init was dropped, not ported) — hand-write it from the reference below, or run zzop-mcp contract config-surface to print the machine-checked vocabulary of every known key.

zzop.config.jsonc
{
  // What to analyze: one or more directory roots. Multiple roots run a
  // cross-layer (multi-tree) analysis.
  "roots": ["."],

  // Or name each tree explicitly (takes precedence over "roots"):
  // "trees": [
  //   { "root": "./api", "sourceId": "api" },
  //   { "root": "./web", "sourceId": "web" }
  // ],

  "packs": {
    // Extra directories of custom DSL rule packs. These MERGE with the
    // bundled packs; a custom pack whose id matches a bundled one wins.
    "extraDirs": ["./zzop-packs"],
    // Whole packs to disable, by pack id.
    "disabled": ["browser"]
  },

  "rules": {
    // "off"                        -> disable the rule
    "typescript/no-explicit-any": "off",
    // "info" | "warn" | "critical" -> override severity
    "sql/nplus1": "warn",
    // object form -> override severity AND drop findings by file path. Each
    // "exclude" entry is a plain substring, or a glob if it has * ? or {}.
    "sql/race-condition-toctou": { "severity": "warn", "exclude": ["legacy/", "**/*.gen.ts"] }
  },

  // Enables git-history-derived signals. Omit to use engine defaults.
  "git": { "recentDays": 30 },

  // Analysis cache directory (omit to disable caching).
  "cacheDir": ".zzop-cache",

  // Files larger than this many bytes skip structural parsing.
  "sizeCap": 500000,

  // "format" / "report.*" / "failOn" are recognized (no unknown-key warning)
  // for forward-compat with the removed JS CLI's config files, but zzop-mcp
  // does not act on them today: output is always the JSON summary on stdout,
  // and the exit code reflects only whether the run itself succeeded (see
  // "Exit codes" below) — gate on a specific severity by parsing the JSON.
  "format": "pretty",
  "failOn": "warn"
}

Config severities normalize to the engine's three levels — off/none/disabled disable a rule; info/note/low → info; warn/warning/medium → warning; error/critical/high → critical.

CommandDescription
zzop-mcp analyze <path>Analyze one repo/tree, print a JSON findings summary. Auto-discovers <path>/zzop.config.jsonc; a config declaring 2+ trees is a guided error pointing at cross instead.
zzop-mcp cross <path>... / cross --config <zzop.config.jsonc>Analyze 2+ trees and print the cross-layer join — zzop's headline. Paths mode needs 2+ roots and ignores any per-tree config; config mode's trees array defines the join.
zzop-mcp endpoint <pattern> <path>... / endpoint <pattern> --config <path>Definitive "is io key X provided/consumed/joined?" query: pattern is substring-matched against every cross-layer io key, answered with a sealed verdict (linked / provided-only / consumed-unprovided / external / unresolved-only / ambiguous / mixed / not-found).
zzop-mcp analyze-envelope <envelope.json>Mode A: the envelope file replaces native parsing entirely for this run; prints the same JSON findings summary shape as analyze.
zzop-mcp validate-envelope <envelope.json>Offline "is this envelope well-formed?" check. Prints {"valid":…,"issues":[…]}; exits 0 valid / 1 invalid.
zzop-mcp validate-rule-pack <pack.json>Offline "does this DSL pack load and every matcher regex compile?" check. Same {valid,issues}/exit-code contract as validate-envelope.
zzop-mcp contract [<name>]No name lists the embedded authoring-contract resources (schemas, guides, the rule catalog, config-surface); a name prints that document's exact bytes to stdout.
zzop-mcp mcpServe MCP over stdio (newline-delimited JSON-RPC 2.0) — what a Claude Code/Desktop config points at.
zzop-mcp version / --versionPrint the binary's version (the same value MCP's initialize reports as serverInfo.version).
zzop-mcp help / --help / -hPrint the usage line plus one elaboration per subcommand, exit 0.

Exit codes are uniform across every subcommand: 0 = ran successfully; 1 = a runtime failure (unreadable path, invalid envelope/config) — for validate-envelope/validate-rule-pack specifically, 1 instead means "ran fine, found the input invalid"; 2 = an argument-shape mistake (missing/extra/flag-shaped argument). There is no severity-gated CI exit code today — failOn is a recognized-but-inert config key (see above); gate a build by parsing the JSON output's findings/crossLayerFindings severities yourself.

Findings

Suppressing & disabling findings

DSL pack rules (the typescript/no-explicit-any, sql/nplus1-style ids) each carry an inline suppress marker — a short comment placed in the flagged code, e.g. // any-ok — that silences just that occurrence; see the rule catalog's "Suppress marker" column for each rule's exact marker. Native analyses, including every cross-layer/* rule, don't support inline suppression — turn them off entirely via config instead.

Disable a native analysis
{
  "rules": {
    "dead-candidates": "off",
    "cross-layer/unconsumed-endpoint": "off"
  }
}

Embedders pass the same ids through disabledRules instead of the rules: { "<id>": "off" } config shape.

Programmatic / embedding

Embedding the engine

Building a tool on top of zzop rather than running it as a CLI? Depend on the zzop-facade / zzop-summary Rust crates directly, or shell out to this same binary's JSON subcommands — either way it's the same engine the CLI drives, JSON in / JSON out. The SDK reference documents the seven functions, the request/output shapes, and multi-repo (cross) analysis.
